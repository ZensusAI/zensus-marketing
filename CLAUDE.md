# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

Marketing site for Zensus (`zensus.app`). The **product app** lives at a separate domain (`app.zensus.app`) and is not in this repo. This repo serves the public landing, pricing, blog, changelog, security, integrations (hub + per-provider sub-pages for Plaid, QuickBooks, HubSpot, and Slack), `/privacy`, `/terms`, and `/subprocessors` pages. One legacy route is permanently redirected at the edge via `vercel.json`, not by a React component: `/features` goes to the homepage. Everything else is a real React route served from this repo. Old pre-migration app paths on the apex (`/runway`, `/forecast`, `/talk`) intentionally return 404; do not add redirects for them, and expect Search Console to list them under "Not found (404)" until Google drops them. Keep any edge redirect single-hop to a 200 page and never redirect to a URL with a `#fragment`; Google Search Console reports fragment targets as "Redirect error".

## Lockfile policy

**This project is pinned to npm.** `bun.lock`, `bun.lockb`, `pnpm-lock.yaml`, and `yarn.lock` are gitignored to prevent cross-tool drift. Do not commit or generate them; always use `npm install`.

## Conventions

- **No em-dashes.** Anywhere. Code, blog content, copy, comments, commit messages. Use commas, parentheses, semicolons, or split sentences instead. The site's voice reads as a series of plain declarative sentences; em-dashes break that rhythm and AI engines quote them awkwardly.
- **`main` is branch-protected.** Ship via PR + GitHub UI merge (squash preferred), not direct push. Direct pushes leave a warning even when an account bypasses the rule. Workflow: branch from `origin/main`, push the branch, open a PR, click "Squash and merge". `main` auto-deploys to Vercel on merge.
- **Write blog content for humans and AI engines.** Use the real Zensus brand name in examples (not placeholders like ACME) so AI crawlers reinforce the entity. Structure prose for plain-text citability: short standalone sentences, lead-in lines into bullet lists, FAQs. See the Blog publishing section below.

## Architecture

**Stack:** React 18 + TypeScript + Vite (SWC) + Tailwind + shadcn/ui + React Router v6 + TanStack Query. Path alias `@` → `src/`.

**Routing (`src/App.tsx`):** Single `BrowserRouter`. Home is eager; every other route (Pricing, Blog, Changelog, Security, Integrations, and per-provider integration pages) is lazy-loaded under a `Suspense` boundary. The catch-all `*` stays last (a comment in the file calls this out). Product-surface redirects live in `vercel.json`, not in React, so hitting `/privacy` etc. never touches the SPA.

**Landing page composition (`src/pages/Index.tsx`):** A deliberate 300ms `<PageSkeleton />` gate before rendering the real page to avoid flash-of-unstyled-content, then hash-scrolls to any `#section` in the URL after mount. Section components live under `src/components/landing/` and are composed top-to-bottom inside `<Index>`.

**Styling:** shadcn/ui (slate base) with CSS variables. `tailwind.config.ts` defines custom animations (`fade-in`, `fade-in-up`, `slide-in-*`, `pulse-glow`, `shine`) that the landing sections rely on; prefer these over adding new one-offs. Fonts are **Geist** (sans/display) and **Geist Mono**, wired through `fontFamily.sans`, `fontFamily.display`, and `fontFamily.mono` in `tailwind.config.ts`. They are **self-hosted**, not loaded from Google Fonts: the variable woff2 files live in `public/fonts/` (one file per family covers all weights, vendored from `@fontsource-variable/geist` + `@fontsource-variable/geist-mono`), `@font-face` declarations are at the top of `src/index.css`, and the sans file is preloaded in `index.html`. This removes a render-blocking cross-origin request to `fonts.googleapis.com`/`fonts.gstatic.com` that delayed first paint on slow connections. Do not re-add a Google Fonts `<link>`.

## SEO and prerender

Every route listed in `src/App.tsx` is also listed in `scripts/prerender.mjs` (static routes plus `/blog/<slug>` from `scripts/blog-slugs.mjs`). `public/sitemap.xml` is generated by `scripts/generate-sitemap.mjs` before the Vite build (static URLs plus blog slugs). The production build runs `vite build` then `scripts/generate-og.mjs` (per-route Open Graph PNGs into `dist/og/`) then `scripts/prerender.mjs` (Puppeteer walks each route and writes `dist/<route>/index.html` so Googlebot sees real HTML on first fetch) then `scripts/indexnow-ping.mjs` (POSTs every sitemap URL to the IndexNow API so Bing and Yandex learn about changes immediately; gated on `VERCEL=1` + `VERCEL_ENV=production` so local builds never ping). The key comes from the `INDEXNOW_KEY` env var on Vercel if set, otherwise from the single `public/<32-hex>.txt` proof file. Pushes to `main` also run `.github/workflows/indexnow.yml`, which POSTs the homepage + sitemap (repository secret `INDEXNOW_KEY`). See `docs/indexnow.md`. When adding a new static route, update `App.tsx`, `scripts/prerender.mjs` (`STATIC_ROUTES`), and re-run `node scripts/generate-sitemap.mjs`. For a new blog post, add `src/content/blog/<slug>.mdx` with exported `meta` (slug must match filename); sitemap, prerender, and OG pick it up automatically.

Per-route `<title>`, description, canonical, and og/twitter tags are emitted by `react-helmet-async` inside each page component. The FAQ section emits the only `FAQPage` JSON-LD block; site-wide Organization / WebSite / SoftwareApplication lives in `index.html`. Breadcrumb JSON-LD is built via `src/lib/structured-data.ts`.

## Blog publishing

**Add a post:** drop `src/content/blog/<slug>.mdx` exporting a `meta` object (slug must match filename). `scripts/blog-slugs.mjs` discovers it; sitemap, prerender, OG, and IndexNow pick it up on the next build. No edits to `App.tsx`, `STATIC_ROUTES`, or any script needed.

**Meta fields.** Required: `title`, `description`, `date`, `category`, `readTime`, `slug`, `ogImage`. Optional but encouraged: `seoTitle` (short string for the `<title>` tag, since `meta.title` can be long for the visible H1 and `BlogPosting.headline`), `ogTitle` / `ogSubtitle` (short for OG cards), `dateModified`, `faqs` (5 entries power both the in-page accordion and `FAQPage` JSON-LD), `tags` (drives the index `?tag=` filter), `featured`, `thumbnail`, `author` (key into `src/lib/authors.ts`, defaults to `ajin`).

**Voice and structure** (optimized for AI citation as well as human reading):
- Lead with `<p id="blog-article-lead" className="text-lg leading-relaxed text-muted-foreground">` and one declarative sentence.
- Use `<p className="mb-2 font-medium text-foreground">Lead-in line:</p>` followed by a `<ul>` of short bullets, so every fact stands alone as a quotable line.
- H2 headings (`##` in MDX) are auto-wired through `CopyHeader`: each heading gets a slugged `id`, hover reveals a link icon, click copies the anchor URL and smooth-scrolls.
- Include 5 `faqs` covering natural People-Also-Ask shapes ("what is X", "how does X differ from Y", "is X safer than Y"). They populate the visible FAQ accordion and the `FAQPage` JSON-LD that targets Google's PAA box.
- Include backlinks to other posts (`/blog/<other-slug>`), product pages (`/integrations`, `/integrations/<provider>`, `/security`, `/subprocessors`, `/pricing`), and authoritative external primary sources. Both internal and outbound links help citation and ranking. When a new post lands, add a reciprocal link from the older post to the new one to build a topical cluster.

**Components** in `src/components/blog/`:
- `BlogCallout` (boxed CTA, typically at the end of a post).
- `BlogFigure` (image + caption, wraps `<figure>` + `<figcaption>`).
- The post and index page chrome (`BlogCard`, `TagFilter`, `TableOfContents`, `MobileToc`, `AuthorCard`, `CopyHeader`, `ReadMoreSection`, `FlickeringGrid`) does not need to be touched per post.

**Illustrations.** Drop PNGs into `public/blog/<slug>/<descriptive-kebab-name>.png`. Reference them with `<BlogFigure>` and **entity-dense alt text** (alt is what AI engines index, so name the entities: "QuickBooks", "MCP server", "OAuth padlock", etc.) plus a caption that restates the section's point. After dropping new images, run `node scripts/optimize-blog-images.mjs`: it scans `public/blog/**/*.png`, resizes anything wider than 1600 px, and re-encodes with high PNG compression. Source PNGs from generative image tools are typically 5–6 MB; the optimizer brings them to ~300–500 KB without visible quality loss.

## Deployment

Deployed via Vercel. `main` auto-deploys to production. Optional **Vercel Production** env var: `INDEXNOW_KEY` (32-char hex, same as `public/<key>.txt`) so IndexNow pings do not rely only on the committed key file. GitHub Actions needs the same value as repository secret `INDEXNOW_KEY` for the IndexNow workflow on `main`.
